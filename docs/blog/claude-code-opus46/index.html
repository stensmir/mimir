Claude Opus 4.6 — The New Flagship Claude Opus 4.6 — Новый флагман

Anthropic just shipped their smartest model yet. Claude Opus 4.6 isn't just an incremental update — it's a genuine step change in what's possible with AI-assisted coding.

A notable addition: 1 million token context window (in beta). This is the first time an Opus-class model offers this — though it's not yet available everywhere and most workflows still use the standard 200K window.

But context size is just the start. Opus 4.6 is specifically tuned for sustained agentic work:

  • Better planning — thinks through multi-step tasks more carefully before diving in
  • Longer focus — maintains coherence across extended coding sessions
  • Smarter debugging — catches subtle issues that previous models missed
  • Large codebase navigation — understands project structure and follows conventions

New capabilities include compaction (Claude can summarize its own context to work even longer), adaptive thinking, and granular effort controls. State-of-the-art on Terminal-Bench 2.0, Humanity's Last Exam, and BrowseComp.

Available as claude-opus-4-6. Pricing stays the same: $5/$25 per million tokens.

Anthropic выпустили свою самую умную модель. Claude Opus 4.6 — это не инкрементальное обновление, а настоящий качественный скачок в возможностях ИИ-ассистента для программирования.

Но размер контекста — только начало. Opus 4.6 специально заточен под длительную агентную работу:

  • Лучшее планирование — тщательнее продумывает многошаговые задачи перед началом
  • Длительный фокус — сохраняет связность в длинных сессиях
  • Умнее дебаг — ловит тонкие баги, которые предыдущие модели пропускали
  • Навигация по большим кодбазам — понимает структуру проекта и следует конвенциям

Новые возможности: compaction (Claude сжимает собственный контекст для более длинных задач), адаптивное мышление и гранулярные настройки усилий. State-of-the-art на Terminal-Bench 2.0, Humanity's Last Exam и BrowseComp.

Доступна как claude-opus-4-6. Цены те же: $5/$25 за миллион токенов.

Agent Teams — Parallel Claude Instances Agent Teams — Параллельные экземпляры Claude

This is the feature that changes how you work with Claude Code. Instead of one Claude doing everything sequentially, you can now spin up a team: one lead agent coordinating multiple teammates working in parallel.

To enable: add this to your settings.json env section:

CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

When to use Agent Teams:

  • Parallel code review — each teammate reviews a different module
  • Independent features — build auth, payments, and notifications simultaneously
  • Competing hypotheses — have teammates explore different approaches to a bug
  • Cross-layer work — frontend, backend, and infra changes in parallel

When NOT to use (expensive!):

  • Sequential tasks where each step depends on the previous
  • Same-file edits — teammates will conflict
  • Simple tasks that don't benefit from parallelism

Display modes: Use Shift+Up/Down to cycle through teammates in-process. For dedicated views, split into tmux panes or iTerm2 tabs.

Delegate mode: Press Shift+Tab to switch to delegate mode where the lead only coordinates and never implements directly.

Plan approval (v2.1.33): Require teammates to submit a plan before implementing — you approve or reject before any code is written.

Quality gates (v2.1.33): New hook events TeammateIdle and TaskCompleted let you run validation when teammates finish work.

Это фича, которая меняет способ работы с Claude Code. Вместо одного Claude, делающего всё последовательно, теперь можно запустить команду: лид-агент координирует нескольких тиммейтов, работающих параллельно.

Чтобы включить: добавьте в секцию env вашего settings.json:

CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

Когда использовать Agent Teams:

  • Параллельный код-ревью — каждый тиммейт ревьюит свой модуль
  • Независимые фичи — auth, платежи и нотификации одновременно
  • Конкурирующие гипотезы — тиммейты исследуют разные подходы к багу
  • Кросс-слойная работа — фронт, бэк и инфра параллельно

Когда НЕ использовать (дорого!):

  • Последовательные задачи, где каждый шаг зависит от предыдущего
  • Правки в одном файле — тиммейты будут конфликтовать
  • Простые задачи без выгоды от параллелизма

Режимы отображения: Shift+Up/Down для переключения между тиммейтами в процессе. Для отдельных окон — tmux panes или вкладки iTerm2.

Режим делегирования: Shift+Tab переключает в режим, где лид только координирует и никогда не имплементирует напрямую.

Согласование плана (v2.1.33): Требуйте от тиммейтов план перед реализацией — вы апрувите или реджектите до написания кода.

Quality gates (v2.1.33): Новые хук-события TeammateIdle и TaskCompleted для запуска валидации когда тиммейты заканчивают работу.

Agent Teams in Practice

Here's a real workflow example. You want to add OAuth to your app — Google, GitHub, and email/password. Instead of doing them one by one:

"Set up OAuth authentication for this app. Have three teammates work in parallel: - Teammate 1: Google OAuth provider - Teammate 2: GitHub OAuth provider - Teammate 3: Email/password with verification Each should create their own files in src/auth/providers/. I'll review before you integrate them into the main auth flow."

The lead coordinates, teammates work independently, and you review each implementation before final integration. What would take an hour sequentially now happens in 15 minutes.

Agent Teams на практике

Вот реальный пример. Вы хотите добавить OAuth — Google, GitHub и email/password. Вместо последовательной реализации:

"Настрой OAuth-аутентификацию для этого приложения. Три тиммейта параллельно: - Тиммейт 1: Google OAuth провайдер - Тиммейт 2: GitHub OAuth провайдер - Тиммейт 3: Email/password с верификацией Каждый создаёт свои файлы в src/auth/providers/. Я проревьюю перед интеграцией в основной auth flow."

Лид координирует, тиммейты работают независимо, вы ревьюите каждую реализацию перед финальной интеграцией. То, что заняло бы час последовательно, происходит за 15 минут.

Memory — Context That Persists Memory — Контекст, который сохраняется

This is subtle but powerful. Claude Code now automatically records and recalls memories as it works. Important decisions, project conventions, your preferences — Claude remembers them across sessions.

No more re-explaining "we use tabs not spaces" or "the API client is in src/lib/api". Claude learns and remembers.

Memory scopes (v2.1.33): The new memory frontmatter field for agents controls where memories are stored:

  • user — persists across all your projects
  • project — specific to this codebase
  • local — only for this working directory

This is a significant change in how Claude Code maintains context. The more you use it, the more it adapts to your style.

Неброская, но мощная фича. Claude Code теперь автоматически записывает и вспоминает важное в процессе работы. Ключевые решения, конвенции проекта, ваши предпочтения — Claude помнит их между сессиями.

Больше не нужно повторять «мы используем табы, не пробелы» или «API-клиент лежит в src/lib/api». Claude учится и запоминает.

Области памяти (v2.1.33): Новое поле memory во frontmatter агентов контролирует, где хранятся воспоминания:

  • user — сохраняется для всех ваших проектов
  • project — специфично для этой кодбазы
  • local — только для этой рабочей директории

Это существенное изменение в том, как Claude Code поддерживает контекст. Чем больше используете — тем лучше он адаптируется к вашему стилю.

Other Notable Changes

  • "Summarize from here" — partial conversation summarization, not just the whole history
  • Skills auto-loading — skills from --add-dir are now loaded automatically
  • TeammateIdle and TaskCompleted hooks — run custom logic when teammates finish (v2.1.33)
  • Agent type restrictions — Task(agent_type) syntax to control which agent types can be spawned
  • Bug fixes — various fixes for agent teams stability, memory persistence, and edge cases

Другие изменения

  • Авто-загрузка skills — скиллы из --add-dir теперь загружаются автоматически
  • TeammateIdle и TaskCompleted хуки — запуск кастомной логики когда тиммейты заканчивают (v2.1.33)
  • Ограничения типов агентов — синтаксис Task(agent_type) для контроля каких агентов можно спавнить
  • Исправления — баги agent teams, персистентности памяти и краевые случаи

What This Means

These updates represent a shift from "AI assistant" to "AI team". Agent Teams parallelize work that used to be sequential. Memory eliminates repetitive context-setting. And Opus 4.6's massive context window means entire large projects fit in the model's working memory.

The combination is powerful: smarter model + team coordination + persistent memory. It's a different way of working with AI — less babysitting, more delegating.

Try it. Especially if you've been frustrated by context limits or sequential bottlenecks. This release addresses both.

Что это значит

Эти обновления — переход от «ИИ-ассистента» к «ИИ-команде». Agent Teams параллелизуют работу, которая раньше была последовательной. Memory устраняет повторяющуюся настройку контекста. А огромное контекстное окно Opus 4.6 означает, что целые большие проекты помещаются в рабочую память модели.

Комбинация мощная: умнее модель + координация команды + персистентная память. Это другой способ работы с ИИ — меньше присмотра, больше делегирования.
